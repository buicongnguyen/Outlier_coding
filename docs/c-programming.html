1. Language and Memory Model

Objects, lvalues, storage duration and linkage, sequence points, integer promotion, strict aliasing, const/volatile/restrict, and where undefined behavior comes from.

Undefined behavior

A construct for which the standard imposes no requirements. Signed overflow, out-of-bounds access, use-after-free, and data races are UB; the optimizer may assume they never happen.

Storage duration

Automatic objects live for their block, static objects for the whole program, allocated objects until freed. Only static and thread-local objects are zero-initialized.

Strict aliasing

An object may not be accessed through a pointer of an incompatible type - except a char pointer, which may alias anything, which is why byte-wise copies are always legal.

const / volatile / restrict

const is read-only intent; volatile forces each access and forbids eliding it; restrict is a no-aliasing promise that lets the compiler keep values in registers.

Declarations right-to-left

const int *p is a pointer to const int; int *const p is a const pointer to mutable int. Reading the declarator from the name outward disambiguates it.

Sequence points

Modifying an object twice, or reading it other than to compute its new value, between sequence points is UB - which is why i = i++ is a bug.

Typical process memory layout (low to high address)

low  +------------------+
     | .text (code)     |  read-only, executable
     +------------------+
     | .rodata          |  string literals, const data
     +------------------+
     | .data            |  initialized globals/statics (copied from image)
     +------------------+
     | .bss             |  zero-initialized globals/statics
     +------------------+
     | heap  --------->  |  malloc/free, grows up
     |        ...        |
     |        ...        |
     |  <--------- stack |  automatic locals + return addrs, grows down
high +------------------+

automatic = lifetime tied to scope     static/global = whole program
allocated (heap) = until you free()     dangling = pointer outliving object

2. Dynamic Memory Management

malloc/calloc/realloc/free semantics, alignment, ownership, leaks, dangling pointers, double free, and use-after-free, plus safe allocation patterns.

Ownership

Every allocation needs exactly one owner responsible for freeing it exactly once. Fuzzy ownership causes leaks, double frees, and use-after-free.

malloc vs calloc

malloc returns uninitialized memory; calloc zero-initializes and guards against nmemb*size overflow. Both can return NULL, so check the result.

realloc carefully

On success realloc may move the block and free the old pointer; assign to a temporary so a NULL return does not lose your only reference.

free is exact

free(NULL) is a safe no-op. Double-freeing or freeing a non-malloc pointer is UB. Setting a pointer to NULL after free turns later misuse into a safe no-op or a clean crash.

Alignment

malloc returns memory aligned for any type (max_align_t). For stronger alignment (SIMD, DMA) use aligned_alloc or posix_memalign.

Overflow-safe sizing

size_t is unsigned and wraps silently, so check count against SIZE_MAX/element_size (or use calloc) before multiplying to avoid a tiny allocation and a heap overflow.

Heap block lifecycle and the classic bugs

p = malloc(n) ──► [ live block ]  ── use p ──►  free(p) ──► [ reclaimed ]
                       |                              |
                       | forget to free               | keep using p
                       v                              v
                    LEAK (unreachable,           USE-AFTER-FREE (UB;
                    never returned)              allocator may reuse memory)

              free(p); ... free(p);   ──►  DOUBLE FREE (UB, heap corruption)

safe habit:  free(p); p = NULL;   // free(NULL) is a no-op, *NULL faults cleanly

3. Systems and Low-Level Programming

File descriptors, syscalls and errno, fork/exec, mmap, signals and async-signal safety, endianness, bit manipulation, and memory-mapped I/O.

File descriptors

Small non-negative integers indexing the open-file table; 0/1/2 are stdin/stdout/stderr. FILE* streams add user-space buffering on top of a descriptor.

errno discipline

Check the documented failure return first, then read errno; it is not cleared on success, so a stale value can be misread.

fork / exec

fork returns 0 in the child and the child's PID in the parent; exec replaces the process image keeping the PID. The common pattern is fork then exec, then wait to reap the child.

Async-signal safety

A handler may interrupt a non-reentrant call, so it may call only async-signal-safe functions (write, _exit). Set a volatile sig_atomic_t flag or write to a self-pipe.

Bit manipulation

Mask with (1u<<n) to test, |= to set, &= ~mask to clear, ^= to toggle. Left-shifting a negative or shifting by >= width is UB; use unsigned types.

volatile and ordering

volatile forces real memory accesses for MMIO and signal flags but gives no atomicity or cross-thread ordering; use _Atomic or fences for concurrency.

fork + exec + wait, and the file-descriptor table

parent                         parent                 child
  |  fd table: 0 1 2 ...          |                      | (copy of fd table)
  |  pid = fork();  ──────────────+──────────────────────+
  |                               | pid > 0 (child PID)   | pid == 0

4. Performance Optimization

Cache locality, access patterns, branch prediction, inlining, false sharing, restrict-enabled optimization, vectorization, and measuring before tuning.

Measure first

Optimization should be driven by a profiler. Cache misses, branch mispredicts, and memory bandwidth often matter far more than raw instruction count.

Cache lines

Memory moves in lines. Sequential, stride-one access reuses whole lines and lets the prefetcher stay ahead; scattered access wastes bandwidth and stalls on misses.

Row-major order

C stores arrays row-major, so the rightmost index should vary fastest in the inner loop to walk memory contiguously.

False sharing

Threads writing different variables that share one cache line cause coherence ping-pong. Pad or align per-thread data to a cache line.

Bound classification

Bandwidth-bound is limited by moving data; compute-bound by arithmetic. The fix differs: improve locality/reduce traffic vs increase parallelism/reduce work.

restrict and inlining

restrict lets the compiler assume no aliasing and keep values in registers; inlining removes call overhead and exposes the body to further optimization.

Memory hierarchy: capacity grows, latency grows

          size            approx latency
registers  ~KB             < 1 ns        <-- keep hot values here
L1 cache   ~32-64 KB       ~1 ns
L2 cache   ~256KB-1MB      ~4 ns
L3 cache   ~8-32 MB        ~15 ns
DRAM       GBs             ~80-100 ns    <-- ~100x an L1 hit
SSD/disk   TBs             10us - ms

goal: raise "work per byte loaded" (arithmetic intensity) so the working
set stays high in the hierarchy - tiling/blocking, contiguous access, SoA.

5. C Libraries and Tooling

String and memory functions and their footguns, formatted I/O safety, pthreads primitives, the build and link pipeline, and sanitizers and profilers.

String footguns

strcpy/strcat have no bounds and overflow; strncpy may not null-terminate; gets is removed. Prefer snprintf and validate lengths.

memcpy vs memmove

memcpy is UB on overlapping regions; memmove handles overlap. memcmp compares raw bytes and is unsafe for structs with padding or floats.

Format-string safety

printf(user_input) lets an attacker control specifiers like %n. Always use a fixed format string: printf("%s", user_input).

Compile then link

The compiler checks syntax and types per translation unit; the linker resolves symbols across objects and libraries. "undefined reference" is a link-time error.

Headers

Include guards or #pragma once make headers idempotent; static inline in a header gives each unit its own copy with no duplicate-symbol errors.

Sanitizers

ASan finds buffer overflows and use-after-free, TSan finds data races, UBSan finds undefined behavior, and Valgrind finds leaks and invalid memory use.
