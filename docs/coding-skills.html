1. Coding

Writing correct, readable code: choosing the right data structure, reasoning about time and space complexity, and handling edge cases before they become bugs.

Correctness first

Code is correct when it meets its specification for all valid inputs, including empty, minimal, maximal, and duplicate cases. Readability is what lets a reviewer verify that quickly.

Big-O complexity

Describes how running time or memory grows with input size n, ignoring constants: O(1), O(log n), O(n), O(n log n), O(n²). Compare algorithms by their growth rate, not their wall-clock time on one input.

Core data structures

Array (index access O(1)), linked list (cheap insert/remove at a known node), hash map (average O(1) lookup), stack (LIFO), queue (FIFO), heap (min/max in O(log n)), balanced tree (ordered ops in O(log n)).

Edge cases

Empty input, single element, all-equal elements, negative numbers, integer overflow, unicode, null/None, and boundary indices (first, last, off-by-one). Enumerate them before writing the loop.

Invariants

A property that holds before and after every loop iteration (for example, "left..right always brackets the answer" in binary search). Stating the invariant is the fastest way to argue a loop is correct.

Stability and in-place

A stable sort preserves the relative order of equal keys (merge sort yes, plain quicksort/heapsort no). In-place means O(1) extra space. Both are common trade-off questions.

Choosing a structure by dominant operation

need fast lookup by key?        ── yes ──>  hash map          O(1) avg
need ordered keys / range scan? ── yes ──>  balanced BST      O(log n)
need min or max repeatedly?     ── yes ──>  heap              O(log n) push/pop
need FIFO processing?           ── yes ──>  queue / deque     O(1) both ends
need LIFO / undo / recursion?   ── yes ──>  stack             O(1) push/pop
mostly index reads, few edits?  ── yes ──>  array             O(1) read, O(n) insert

Big-O at a glance (n = 1,000 items, log scale)

O(1)         1 op          ▏          hash lookup, array[i]
O(log n)     ~10 ops       █          binary search
O(n)         1,000         ███        one pass over the data
O(n log n)   ~10,000       ████       merge sort, sorting in general
O(n²)        1,000,000     ██████     nested loops: compare every pair

each extra █ ~ one more zero — the gap explodes as n grows

Why binary search is O(log n)

find 61 in 9 sorted elements — probe the middle, drop half each step

step 1   [  4   8  15  16 (23) 42  61  77  90 ]    23 < 61 → drop left half
step 2                       [ 42 (61) 77  90 ]    61 = 61 → found

window: 9 → 4 → 2 → 1        steps ≈ log2(n), the loop invariant is
                             "if present, target is inside [left..right]"

2. Debugging & Repair

Turning a failure into a fix: read the stack trace, reproduce, bisect to isolate, find the root cause, then write a minimal fix plus a regression test that would have caught it.

Stack trace

The chain of active calls at the moment of failure. The innermost frame (usually printed first) shows where the error was raised; walk outward to find the frame in your own code that supplied the bad value.

Reproduce, then minimize

A bug you can trigger on demand is half fixed. Shrink the failing input and the code path until removing anything else makes the failure disappear - that minimal repro usually points straight at the cause.

Bisecting

Binary search over history or over code: git bisect finds the first bad commit in O(log n) steps; commenting out half the suspect code does the same for a code path.

Root cause vs symptom

Patching where the crash happens (for example a null check at the crash site) often hides the real defect upstream. Ask "why was this value wrong?" repeatedly until the answer is a design or logic decision.

Minimal fix

The smallest change that corrects the root cause without reworking unrelated code. Small diffs are easier to review, safer to ship, and easier to revert.

Regression test

A test that fails on the buggy code and passes after the fix. It documents the bug and guarantees the same defect cannot silently return.

3. Compilers & Programming Languages

How source text becomes executable code: lexing, parsing, the AST, semantic analysis and type checking, lowering to an intermediate representation, optimization passes, and code generation.

Lexing (tokenizing)

Splits raw source text into tokens - identifiers, keywords, literals, operators - and discards whitespace and comments. Output: a flat token stream.

Parsing

Consumes the token stream according to the language grammar and builds a tree. Syntax errors ("unexpected token") are reported here.

AST

Abstract Syntax Tree: the parsed program as nested nodes (function, if, call, literal), stripped of punctuation. The data structure most tools - linters, formatters, transpilers - actually work on.

Type checking

Part of semantic analysis: verifies that operations match operand types, resolves names against scopes, and infers types where the language allows. Catches whole classes of bugs before the program runs.

IR (intermediate representation)

A simplified, machine-independent form of the program (for example LLVM IR, often in SSA form). It decouples front ends (languages) from back ends (CPUs) so optimizations are written once.

Optimization & codegen

Passes over the IR: constant folding, dead-code elimination, inlining, loop transforms. Codegen then selects machine instructions, allocates registers, and emits the final binary or bytecode.

From source text to AST: x = 1 + 2 * y

source     x = 1 + 2 * y

lexer      IDENT(x)  '='  INT(1)  '+'  INT(2)  '*'  IDENT(y)     flat token stream

parser         =
              / \
             x   +           * binds tighter than +, so the tree
                / \          encodes operator precedence that the
               1   *         flat token stream cannot express:
                  / \        the program means x = 1 + (2*y),
                 2   y       never (1+2) * y

4. Porting & Migration

Moving code across languages, runtimes, or versions while preserving behavior: shim the old API, prove parity with tests, and roll out incrementally so you can retreat at any step.

Porting vs migration

Porting moves code to a new language, platform, or runtime. Migration moves a system to a new version, schema, or service. Both share the same core risk: silent behavior change.

Semantic gaps

The dangerous differences hide in semantics, not syntax: integer width and overflow, string encoding, floating-point rounding, division/modulo of negatives, sort stability, evaluation order, and timezone handling.

API shim / adapter

A thin compatibility layer that exposes the old interface but delegates to the new implementation. Callers keep working unchanged while the internals move underneath them.

Behavior-parity testing

Run the same inputs through old and new implementations and diff the outputs (golden-master / characterization tests). It captures actual behavior - including bugs callers depend on - not just documented behavior.

Strangler fig pattern

Put a routing facade in front of the legacy system, then move one capability at a time to the new system until the old one has no traffic left and can be deleted. Avoids a big-bang rewrite.

Incremental rollout

Feature flags, canary releases, and shadow traffic (send requests to both, serve from old, compare). Each limits blast radius and gives an instant rollback path.

Strangler-fig migration flow

step 1: all traffic to legacy        step 2: facade routes per feature

clients ──> +--------+               clients ──> +--------+──> legacy (B, C)

5. Tooling

The infrastructure around the code: build systems, linters, formatters, package managers, continuous integration, and profilers. Good tooling turns "works on my machine" into "works everywhere, provably".

Build systems

Make, Gradle, Bazel, MSBuild: they model targets and dependencies as a graph and rebuild only what changed (by timestamp or content hash). Correct incremental builds require accurately declared dependencies.

Linters

Static analysis of source or AST for likely bugs and style violations: unused variables, unreachable code, suspicious comparisons. They find problems without running the code.

Formatters

Rewrite code layout (whitespace, wrapping, quotes) to one canonical style without changing behavior. They end style debates and shrink diffs to real changes.

Package managers & lockfiles

npm, pip, cargo, NuGet resolve and fetch dependencies. The manifest says what you want (ranges); the lockfile pins the exact resolved versions so every machine and CI run installs identical bytes.

Continuous integration

Every push triggers an automated pipeline: build, lint, test, package. Failures surface within minutes of the change that caused them, while the diff is small and fresh in the author's mind.

Profilers

Measure where time and memory actually go. Sampling profilers periodically capture stacks (low overhead); instrumenting profilers time every call (exact but slower). Always measure before optimizing.
