1. Repository Management

Creating and cloning repositories, the working tree / index / HEAD model, remotes, .gitignore, tags, bare repos, submodules, and object cleanup.

The three trees

Git tracks the working directory, the staging area (index), and HEAD (the last commit). Most commands move changes between these three areas.

init vs clone

init creates a new empty repo (a .git directory); clone copies an existing remote including its full history and sets up an origin remote.

Remotes

A remote is a named reference (like origin or upstream) to another copy of the repo. You interact with it via fetch, pull, and push.

.gitignore

Lists patterns of untracked files Git should ignore. A file already tracked keeps being tracked until you git rm --cached it.

Tags

A tag names a specific commit. Annotated (or signed) tags carry a tagger, date, and message and suit releases; lightweight tags are just a name. Tags are not pushed automatically.

Bare repos & submodules

A bare repo has no working tree and serves as a shared remote. A submodule embeds another repo at a pinned commit inside a parent repo.

The three trees and where commands move changes

working directory        staging area (index)        HEAD (last commit)
   your edited files   ── git add ──►   staged snapshot  ── git commit ──►  new commit
        ▲                                    ▲                                   |
        └──── git restore  ───────────┘                                   |
        └──── git restore --source=HEAD (or checkout) ──────────────────────────┘

remote (origin):   git fetch ──► remote-tracking refs      git push ──► publish commits

2. Branching Strategies

Branches as pointers, HEAD, fast-forward versus three-way merges, rebase, detached HEAD, and models such as trunk-based development and Git Flow.

Branch is a pointer

A branch is a lightweight movable reference to a commit; creating and switching branches is cheap because no files are copied.

HEAD

HEAD points at the currently checked-out commit, usually through a branch. When it points directly at a commit you are in detached-HEAD state.

Fast-forward vs three-way

Fast-forward just advances the pointer when there is no divergence; a three-way merge combines diverged branches via their common ancestor into a merge commit.

Merge vs rebase

Merge preserves true history with a merge commit; rebase replays commits onto a new base for a linear history but rewrites commit IDs, so never rebase shared branches.

Trunk-based

Everyone integrates small changes into one main branch frequently using very short-lived branches, which pairs well with feature flags and CI.

Git Flow

A heavier model with long-lived main and develop branches plus feature, release, and hotfix branches; it suits scheduled releases.

Fast-forward vs three-way merge

FAST-FORWARD (no divergence)        THREE-WAY MERGE (both moved)

main ─A─B                            main ─A─B───────M   ← merge commit (2 parents)
        \                                   \       /
feature  C─D                        feature  C─────D
                                    base = A (common ancestor); conflicts where
merge → main just moves to D        both A..B and A..D changed the same lines

3. Collaboration Workflows

Fetch, pull, and push; forks and upstream; pull requests and review; resolving conflicts; protected branches; and staying in sync with the mainline.

fetch vs pull

fetch only downloads remote changes into remote-tracking branches; pull fetches and then merges (or rebases) into your branch. push publishes local commits.

Fork & upstream

A fork is a server-side copy under your account. Contributors add an "upstream" remote pointing at the original to keep their fork in sync.

Pull request

A request to merge a branch, enabling review, discussion, and automated checks before integration. It is a platform feature, not a Git command.

Conflict resolution

Conflicts happen where both sides changed the same region. Edit the file to the intended result, remove the markers, stage, and commit.

Protected branches

Rules on a branch such as no force-push, no deletion, and required reviews or passing CI before a merge is allowed.

Stay in sync

Merging or rebasing from main frequently surfaces conflicts in small increments instead of one large painful merge later.

Fork / upstream / origin flow and fetch vs pull

upstream (original) ──fetch──► your remote-tracking refs ──merge──► local main
      ▲                                                                 |
      │ pull request                                                    │ push
      │                                                                 v
   origin (your fork)  ◄──────────────────────────────────────────  local branch

fetch = download only (safe)      pull = fetch + merge/rebase into current branch

4. Core Git Operations

Commit and amend, reset (soft / mixed / hard), revert, cherry-pick, stash, reflog, bisect, and reading log and diff.

commit & amend

commit records the index as a new snapshot; --amend replaces the most recent commit. Avoid amending commits already pushed to a shared branch.

reset modes

soft moves HEAD only; mixed also resets the index (default); hard resets index and working tree, discarding changes. hard is destructive.

revert vs reset

revert adds a new commit that undoes another and is safe for public history; reset moves branch pointers and is best for local history.

cherry-pick & stash

cherry-pick applies one specific commit onto the current branch; stash saves uncommitted changes and cleans the working tree so you can switch context.

reflog

Records where HEAD and branches have pointed, so commits "lost" after a bad reset or rebase are usually recoverable until it expires.

bisect

Binary-searches history between a known-good and known-bad commit to isolate the change that introduced a bug in about log2(n) tests.

git reset modes: what each one touches

                         moves HEAD    resets index    resets working tree
git reset --soft        yes            no                 no      (changes stay staged)
git reset --mixed       yes            yes                no      (changes stay, unstaged)
git reset --hard        yes            yes                yes     (changes DISCARDED)

undo the last commit but keep the work staged:   git reset --soft HEAD~1
unstage a file (keep edits):                     git restore --staged 
recover after a bad reset:                       git reflog  →  git reset --hard

5. Best Practices

Atomic commits and clear messages, never rewriting shared history, secrets hygiene, small pull requests, signing, hooks, large-file storage, and safe force pushes.

Atomic commits

One logical, self-contained change per commit that builds and could be reverted on its own, making review, revert, cherry-pick, and bisect effective.

Good messages

A concise imperative summary line, a blank line, then a body explaining what and why when needed. History and blame become far more useful.

Do not rewrite shared history

Rewriting published commits (force-push) breaks collaborators. Rewrite freely on local work; prefer revert once others may have based work on it.

Secrets hygiene

Never commit credentials; history retains them even after deletion, so treat any committed secret as compromised and rotate it. Prevent with .gitignore and scanning.

Small PRs

Small, focused pull requests get thorough reviews, catch more bugs, and merge sooner than large ones that receive shallow review.

Hooks, LFS, signing

Hooks run checks on events (pre-commit, pre-push); LFS stores large binaries as pointers; signed commits/tags prove provenance.

Rewrite-safety rule of thumb

is the commit already pushed / could others have it?
        |                                   |
        no                                  yes
        |                                   |
        v                                   v
   rewrite freely:                     do NOT rewrite:
   amend, rebase -i, reset             use git revert (adds an undo commit)
                                       if you MUST: git push --force-with-lease
                                       (refuses if the remote moved unexpectedly)
